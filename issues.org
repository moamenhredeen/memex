* DONE [#A] Move key dispatch into each view (per-view key dispatch refactor)
  :PROPERTIES:
  :ID: per-view-key-dispatch
  :END:

  Today every keystroke goes through one ~on_key_down~ at ~Memex::render~
  (src/app.rs:1648) and one shared ~KeymapSystem~ with one shared
  ~LayerStack~. Pane focus and keymap layers are kept in sync by hand in
  ~open_item_in_split~ / ~close_split~ / ~focus_pane~ / ~switch_to_item~
  (src/app.rs:1108–1252). When gpui focus diverges from
  ~self.focused_pane~ (e.g. clicking a pane), keys resolve against the
  wrong layer set and either run the wrong view's command or get silently
  dropped by the ~!is_editor()~ guards at src/app.rs:1710/1726/1733.

  Target model: each view owns its own ~FocusHandle~ and its own
  ~KeymapSystem~, and handles its own ~on_key_down~ with
  ~track_focus(...)~. gpui routes key events only to the focused view, so
  the keymap is scoped naturally. The app shell handles only truly
  app-wide shortcuts, registered as gpui ~Action~ s.

  Deliverables:
  - ~EditorView~, ~PdfView~, ~GraphView~ each own focus + keymap + key
    handler.
  - ~Memex~ no longer has an ~on_key_down~, no ~focused_pane~ layer
    choreography, no ~KeymapSystem~ field (or keeps a minimal one for
    window-level bindings only).
  - ~ActiveItem::keymap_layers~ / ~deactivate_layers~ (src/pane.rs:92,
    101) removed.
  - Layer-mutation variants of ~ItemAction~ (~ActivateLayer~,
    ~SetVimEnabled~, ~SyncVimFlags~, src/pane.rs:54–58) removed; each
    view mutates its own keymap directly.
  - Clicking a pane switches focus (and thus key dispatch) with no
    additional app logic.

* TODO [#A] Vault: org-roam-style knowledge graph
  :PROPERTIES:
  :ID: vault-knowledge-graph
  :END:

  Direction chosen: flat notes folder, ID-based filenames, frontmatter
  with tags / status / date / aliases, dedicated folders for
  attachments and journal. Navigation is links + backlinks + search,
  not folder trees. No nested tags, no path-based wikilinks, no file
  explorer for notes (a notes list that filters by tag/title is
  enough). Hierarchy is written as MOC notes.

  Folder layout (fixed, created on vault init):
  #+begin_src
  my-vault/
    notes/              # flat, ID-based filenames
    attachments/        # PDFs, images, future drawings
    journal/            # daily notes, YYYY-MM-DD.md
    .memex/             # per-vault cache, config
  #+end_src

  ID scheme: ~YYYYMMDDTHHMMSS-XXXX~ (UTC timestamp + 4 chars base36).
  20 chars. Sortable chronologically in the filesystem. Short random
  suffix makes fast-capture and multi-device git merges collision-
  free. Filename does NOT encode title or tags — those live in
  frontmatter so edits don't rename files.

  Frontmatter schema (YAML):
  #+begin_src yaml
  ---
  id: 20260423T142301-k3n8
  title: Unit testing
  created: 2026-04-23T14:23:01Z
  tags: [testing, engineering]
  aliases: []
  status: draft      # draft | active | archived (free-form string, recommended set)
  ---
  #+end_src

  Wikilinks:
  - ~[[title]]~ resolved via a title→ID index.
  - ~[[alias]]~ resolved via aliases in frontmatter.
  - ~[[id:20260423T142301-k3n8]]~ resolved directly (rename-safe).
  - Rename-safe in practice: titles are mutable, IDs are not.
    Renaming a title updates frontmatter but does NOT rename the
    file, so no other note needs to be rewritten.

  Scope is split into three phases. Phase 1 is the foundation
  (ID/frontmatter/layout/index); Phase 2 surfaces it in the UI
  (backlinks panel, tag search, full-text search); Phase 3 is
  polish (rename safety, MOC helpers, attachments handling,
  registry management).

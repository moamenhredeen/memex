#+title: Issues

* DONE [#A] Move key dispatch into each view (per-view key dispatch refactor)
  :PROPERTIES:
  :ID: per-view-key-dispatch
  :END:

  Today every keystroke goes through one ~on_key_down~ at ~Memex::render~
  (src/app.rs:1648) and one shared ~KeymapSystem~ with one shared
  ~LayerStack~. Pane focus and keymap layers are kept in sync by hand in
  ~open_item_in_split~ / ~close_split~ / ~focus_pane~ / ~switch_to_item~
  (src/app.rs:1108–1252). When gpui focus diverges from
  ~self.focused_pane~ (e.g. clicking a pane), keys resolve against the
  wrong layer set and either run the wrong view's command or get silently
  dropped by the ~!is_editor()~ guards at src/app.rs:1710/1726/1733.

  Target model: each view owns its own ~FocusHandle~ and its own
  ~KeymapSystem~, and handles its own ~on_key_down~ with
  ~track_focus(...)~. gpui routes key events only to the focused view, so
  the keymap is scoped naturally. The app shell handles only truly
  app-wide shortcuts, registered as gpui ~Action~ s.

  Deliverables:
  - ~EditorView~, ~PdfView~, ~GraphView~ each own focus + keymap + key
    handler.
  - ~Memex~ no longer has an ~on_key_down~, no ~focused_pane~ layer
    choreography, no ~KeymapSystem~ field (or keeps a minimal one for
    window-level bindings only).
  - ~ActiveItem::keymap_layers~ / ~deactivate_layers~ (src/pane.rs:92,
    101) removed.
  - Layer-mutation variants of ~ItemAction~ (~ActivateLayer~,
    ~SetVimEnabled~, ~SyncVimFlags~, src/pane.rs:54–58) removed; each
    view mutates its own keymap directly.
  - Clicking a pane switches focus (and thus key dispatch) with no
    additional app logic.

* DONE [#A] Editor: scrollbar + follow-cursor [4/4]
  :PROPERTIES:
  :ID: editor-scrollbar-follow-cursor
  :END:

  The editor has no scrollbar and doesn't scroll to keep the cursor in
  view. Typing past the bottom of the viewport hides the cursor. Scroll
  wheel works (src/editor/view.rs:232) but there's no visual indicator
  of scroll position or document size.

  PDF already has a working scrollbar at src/pdf/scrollbar.rs
  (~285 lines). It reads ~total_height~/~scroll_offset~ from
  ~PdfState~ and stores drag state on ~PdfState.drag_state~. The
  drawing, hit-testing, drag, and track-click logic are generic —
  only the data source is ~PdfState~-specific. Good candidate for
  reuse.

* DONE [#B] Clicking a pane does not switch focus / keymap layers
  :PROPERTIES:
  :ID: click-focus-bug
  :END:

  Reported symptom that motivated [[id:per-view-key-dispatch]]: with
  editor + PDF split open, clicking on the editor leaves
  ~self.focused_pane = Right~ and leaves the ~pdf~ layer on top of the
  stack, so typing in the editor either runs PDF commands or is
  silently dropped.

  [[id:per-view-key-dispatch]] fixes this structurally. If that refactor
  is deferred, the minimum fix is an ~on_mouse_down~ on each pane
  wrapper calling ~Memex::focus_pane(side, ...)~ — ~focus_pane~ already
  swaps layers correctly (src/app.rs:1200). ~20 lines. Leave this
  ticket open until the refactor lands or this minimum fix is applied.
